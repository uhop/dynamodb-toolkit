# dynamodb-toolkit — v3 Survey

> **Status:** survey only — not a design doc. Maps the current toolkit's surface against AWS JS SDK v3 to inform a future ESM + TS rewrite.
> **Date:** 2026-04-15.
> **Companion design references:**
>
> - The REST surface in `helpers/KoaAdapter.js` was designed against Eugene Lazutkin's _Web Applications: the modern API design_ (`~/Open/articles/design/web-apps-client-server-api-design.md`, ©2012–2018). The article defines **functional requirements** for a sane REST API, not a fixed naming scheme. §4 below distills those requirements and lists the helper functions the toolkit should provide so that implementing concrete endpoints (with project-specific names, codes, and envelopes) is trivial — and in many cases automatable.

## 1. Conceptual model (current toolkit)

Three data shapes flow through the system, with the `Adapter` brokering between them:

| Shape     | What it is                                                   | Marker class    | When you see it                                                                 |
| --------- | ------------------------------------------------------------ | --------------- | ------------------------------------------------------------------------------- |
| user-land | Application JSON (whatever your domain wants).               | none            | Caller-side input/output. Run through `prepare()` / `revive()`.                 |
| `raw`     | DocumentClient-style JSON-with-Sets (no `{S:..}` envelopes). | `Adapter.Raw`   | Internal representation when `isDocClient`. Bypasses `prepare()`/`revive()`.    |
| `db-raw`  | DynamoDB wire format (`{S}`/`{N}`/`{NS}`/`{M}`/...).         | `Adapter.DbRaw` | Internal representation when using raw `AWS.DynamoDB`. Bypasses conversion too. |

Other foundational ideas:

- **Adapter = one logical entity (usually one table).** Constructor takes `client`, `table`, `keyFields`, plus optional `specialTypes`, `searchable`, `searchablePrefix`, `projectionFieldMap`, `indirectIndices`, `converter`, `converterOptions`, and the user hooks below.
- **`isDocClient`** is detected once via `typeof client.createSet == 'function'` (v2-only). All conversion branches pivot on it.
- **User hooks** (override or pass via constructor): `prepare(item, isPatch)`, `revive(rawItem, fields)`, `prepareKey(key, index)`, `restrictKey(rawKey, index)`, `prepareListParams(item, index)`, `updateParams(params, options)`, `async validateItem(item, isPatch)`, `async checkConsistency(batch)`.
- **Batch objects** = `{action, params}` (or `{action, adapter, params}` for `makeGet`). Generated by `make*()`, consumed by `applyBatch` / `applyTransaction` / `getTransaction`. Action is one of `'get'|'check'|'put'|'patch'|'delete'`.
- **Transaction upgrade.** Every CRUD method asks `checkConsistency(batch)`. If it returns extra batch items, the single op is upgraded to `transactWriteItems` covering both the write and the consistency checks.
- **Patches.** `prepareUpdate(patch, deleteProps, params, separator='.')` builds `UpdateExpression`. `__delete: ['paths']` and `__separator: '.'` keys on a patch object are extracted by `makePatch()`. Dotted paths (`'meta.tag'`) become nested `SET`/`REMOVE`; pure-digit segments are treated as array indices and not aliased.
- **Indirect indices.** `indirectIndices: {IndexName: 1}` declares a GSI that projects keys only. Affected reads (`getByKey`, `getByKeys`, `scanAllByParams`, `getAllByParams`) do a second batchGet against the base table via `readOrderedListByKeys`. `ignoreIndirection: true` skips this.
- **Searchable fields.** `searchable: {field: 1}` plus `searchablePrefix` (default `'-search-'`) → `prepare()` writes a lowercase `<prefix><field>` mirror; `filtering()` builds `contains(...)` `FilterExpression` chains over the mirror columns. Substring filtering, case-insensitive by default.
- **Technical-field convention.** Fields whose names start with `-` (e.g., `-t`, `-search-name`) are toolkit-internal and stripped by typical `revive()` implementations. Not enforced by the library; the convention is set by `tests/routes.js`.
- **`returnRaw` modes.** Per-call: `'raw'` returns an `Adapter.Raw`, `'db-raw'` returns an `Adapter.DbRaw`, falsy runs `revive()`. Used to pipe reads back into writes without re-invoking `prepare`/`revive`.

### 2.1 Cross-cutting mechanisms (current)

- **Chunking limits.** Writes 25 (BatchWriteItem ceiling), gets 100 (BatchGetItem ceiling), moves 12 (each move emits a put + delete = 24 actions ≤ 25 transaction ceiling). With v3's transaction limit raised to **100 actions**, move chunks could grow to 50.
- **Retry surface.** Custom backoff is invoked only for `UnprocessedItems`/`UnprocessedKeys`. SDK transport-level retries (5xx, throttles) sit underneath.
- **Expression composition.** Every helper that touches `params` namespaces its aliases (`#upk0`, `:upv0`, `#sr0`, `:flt0`, `#pj0`, `#k0`...) and reuses any pre-existing `ExpressionAttributeNames`/`Values`. `cleanParams()` post-processes to drop unused.
- **Empty-attribute handling.** Current `Converter` supports `convertEmptyValues`. v3 default is to **throw** on `undefined` unless `removeUndefinedValues: true` is passed at `DynamoDBDocumentClient.from(...)`. **This is the most-cited v2→v3 gotcha.**

| Project               | What it does                                                                      | Overlap with toolkit                                                                                                                 |
| --------------------- | --------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| `dynamodb-toolbox` v1 | Schema-driven entities (single-table design), typed queries, expression composer. | High overlap with patch/filter/projection builders, but at a higher abstraction level (you give up generic maps for typed entities). |
| `electrodb`           | Single-table-design ORM with fluent expression builder and typed entities.        | Same as above.                                                                                                                       |
| `dynamodb-onetable`   | Single-table ORM with its own expression mini-language.                           | Same as above.                                                                                                                       |
| `typesafe-dynamodb`   | Type-level wrappers around v3 commands. Doesn't add runtime; tightens types only. | Useful as a layering option for the toolkit's typed-Item story.                                                                      |
| `aws-sdk-client-mock` | Test doubles for v3 (mocks the middleware stack per client). AWS-team-endorsed.   | Replaces the Postman+Koa test loop.                                                                                                  |

The toolkit's market position remains: **schemaless, REST-shaped, no runtime deps, ergonomic patches/filters/projections, mass ops with backoff.** The single-table ORMs are not a drop-in replacement.

## 4. REST functional requirements (from the design article)

The article enumerates capabilities a sane REST layer should expose, not concrete names or status codes. Treat it as guidance: the toolkit's job is to provide **helpers that make each requirement trivial to implement** — and, where the requirement is fully mechanical (CRUD, paging, subsetting), to provide a default implementation the consumer can adopt or override. Concrete naming (envelope keys, prefixes, status codes) is a **per-project policy choice**, configurable on the helper.

### 4.2 Naming and policy knobs (configurable, not prescribed)

The article's specific names are illustrative. The v3 toolkit should expose these as configuration so each project can pick its own house style without forking helpers:

- **Meta-property prefix** on the wire (default `_`): governs `_delete`, `_separator`, `_findOne`, `_findAll`, `_array`, etc.
- **DB-internal field prefix** (default `-`): governs `-search-*`, `-t`, etc. Kept distinct from the wire prefix because it's hidden by `revive()`.
- **Pagination envelope keys**: `items` vs `data`, `total` vs `count`, `last` vs `done`, etc. — supplied as a small mapper or a config object.
- **Status code map**: `{miss, validation, consistency, throttle, transient, internal}` → HTTP code. Default sane values; consumer overrides per project policy.
- **Error envelope shape**: a builder function `(err, ctx) => body` so the consumer fully owns it. Toolkit ships a default builder matching the article's recommendation (code + message + optional severity / retryable / errorId / details).
- **Method-prefix character** (default `-`): the disambiguator between IDs and method names on collection/item URIs.

### 4.3 What the Postman collection actually tests

The suite at `tests/Unit test dynamodb-toolkit.postman_collection.json` is **40 sequential requests** against `tests/routes.js` loaded with the Star Wars planets fixture (`tests/data.json.gz`, 61 items). Each request's embedded script asserts response status, body keys, item counts, and sort order. This is the de facto contract for what the REST layer must do; the v3 rewrite must reproduce it (names and codes are still configurable).

Phases (order matters; each depends on the previous):

1. **Setup** — `DELETE /` (clear), `GET /` (verify empty: `{data: [], total: 0, offset: 0, limit: 10}`), `PUT /-load` → `{processed: 61}`.
2. **List with pagination / sort / filter / subsetting** — `GET /` (page 1, `total: 61`), `GET /?offset=10`, `GET /Alderaan?fields=name,climate,terrain,diameter`, `GET /-by-names?names=Hoth,Endor,Bespin,XX&fields=name,diameter` (missing `XX` silently dropped — no `404`; response is a **plain array**, not a paginated envelope), `GET /?sort=-name&fields=name` (single-field descending), `GET /?sort=-name&fields=name&filter=tooine` (filter + sort; `total: 2`), `GET /?sort=-name&fields=name&filter=a` + `&offset=10` (filter + pagination).
3. **Mass mutation by filter** — `PUT /-clone/?filter=toonie` → `{processed: 2}`, `DELETE /?filter=toonie` → `{processed: 4}` (originals + clones).
4. **Single-item CRUD** — `POST /` (create `AAA Fake`), `GET /:name?consistent=1` (strong read), `PUT /:name` (full replace with nested `config` object), `PATCH /:name` (nested patch using `.` separator and `__delete: ['config.b']`), `PATCH /:name` (nested patch using `__separator: '/'` — exercises both separators), `GET ...?fields=config.a,name` (subset returns `{name, config: {a}}`), `GET ...?fields=diameter,config.c,config.b,config.a` (absent `config.b` silently dropped from the subsetted response).
5. **Clone / move / 404** — `PUT /:name/-clone` creates `AAA Fake COPY` via the routes' default `augmentCloneFromContext` (` COPY` suffix). Verified with `GET`. `DELETE` then `GET` → **`404`**. `PUT /:name/-move` renames; `GET` on the original → **`404`**, `GET` on the new name → `200`. Final `DELETE` + `GET` → **`404`**.
6. **Mass ops by name list** — `PUT /-clone-by-names/?names=Cerea,Endor` → `{processed: 2}`. `GET /-by-names?names=...` returns a plain array, alphabetically sorted. `DELETE /-by-names?names=...` → `{processed: 2}`. Verification `GET /-by-names` returns `[]` (empty array, not `404`). `PUT /-move-by-names/?names=...` → `{processed: 4}` (2 puts + 2 deletes).
7. **Cleanup** — final `DELETE /` → `{processed: 59}`.

Concrete facts the survey must preserve:

- **Pagination envelope keys: `{data, total, offset, limit}`.** No `next`/`previous`/`last`; consumers compute end-of-list as `offset + data.length >= total`. The utility `paginateList` / `paginateListNoLimit` accept a `needTotal: false` flag that produces `{data, offset, limit}` (no `total`), but `Adapter.getAllByParams` doesn't plumb that through — every Koa response in the Postman suite has `total`. **This is a designer-time choice**, not a client-facing switch: the API designer decides whether `total` is worth the `Select: COUNT` scan, ideally by passing `options.needTotal` at the Adapter call site or setting it per-route. A designer _may_ choose to expose a query parameter, but the toolkit should not ship one by default.
- **`-by-names` and `-clone-by-names` / `-move-by-names` / `-clone` / `-move` / `-load` / `-delete-by-names`** are the current OOP-method URIs. Default method-prefix character is `-`.
- **Query params in use:** `offset`, `limit` (implicit default 10), `sort` (`field` or `-field`; single-field), `fields` (inclusion only, dotted paths, nested via `.`), `filter` (substring over `searchable` fields), `names` (comma-separated for `*-by-names`), `consistent` (`1`/`true`/`on`/`yes` per `isTrue.js`), `force` (for `PUT` item replace, `/-clone`, `/-move`).
- **Patch body supports both `__delete` (array of paths) and `__separator` (string)**, proven by the collection using both `.` and `/`. The rename to `_delete`/`_separator` (plus configurable prefix) must preserve this duality.
- **`?filter=` is already on writes**, not just reads — `DELETE /?filter=` and `PUT /-clone/?filter=` are tested. The "PATCH by criterion" addition in §4.1 #4 is an extension of the existing pattern, not net new.
- **DELETE is idempotent on the wire** — returns `204` regardless of prior existence. `404` is only returned by single-item `GET`/clone/move where the _read_ missed.
- **Writes return an empty body with `204`**, never echoing the item back. Mass operations return `{processed: N}`.
- **Missing keys in a `-by-names` request are silently dropped** (no `404`, no error). The response array contains only the items that existed.

### 4.4 What "automatable" means here

For a typical CRUD entity the consumer should be able to write, in pseudocode:

```js
const koa = makeKoaAdapter({
  adapter,
  policy: { metaPrefix: '_', envelope: 'items', statusCodes: {...}, errorBody: ... },
  routes: 'standard'   // mounts /, /:key, /-by-names, /:key/-clone, etc.
});
router.use('/planets', koa.router());
```

…and get every requirement in §4.1 wired up with no per-endpoint glue. Custom endpoints (the OOP-style methods of #2) drop in as one-liners using the same parsers (`parseFields`, `parseSort`, `parseFilter`, `parsePatch`) and the same envelope/error helpers.

## 6. Proposed future functionality (input list for the design doc)

Not decisions — items the design doc must rule on, with the verdict from §2 / §4 attached where it's already obvious.

### 6.1 Architecture and packaging

- **ESM-only**, `"type": "module"`, native `import`/`export`.
- **Source layout: `src/` with functional subfolders — not flat, but don't overdo it.** The current top-level layout (`Adapter.js`, `helpers/`, `utils/` — the last one a 40-file flat dump) is explicitly what we're moving away from. Rules, in order:
  1. **Main entry file lives directly in `src/`** (e.g., `src/index.js` + `src/index.d.ts`) — not inside a subfolder.
  2. **Group related helpers by topic.** Each topic gets a folder named after it: `src/adapter/`, `src/koa/` (and any other framework wrappers), `src/expressions/` (patch / projection / filter / condition builders), `src/batch/` (applyBatch, applyTransaction, getBatch, getTransaction, backoff), `src/mass/` (writeList, deleteList, copyList, moveList, readList family, pagination), `src/paths/` (getPath/setPath/deletePath/applyPatch/normalizeFields/subsetObject).
  3. **Collapse single-file topics to a bare file.** If a topic has only one file, place it at `src/<topic>.js` with its `.d.ts` sidecar (e.g., `src/sleep.js`, `src/seq.js`, `src/random.js`). Promote to a folder only when a second file joins.
  4. **`src/utils/` is reserved for genuinely cross-cutting helpers**, not a dumping ground. If a helper fits one of the topic folders, it goes there.
  5. A folder with an `index.js` that re-exports a single file is worse than the bare file — default to the simpler layout.
- **`.js` sources with hand-written `.d.ts` sidecars — not true TypeScript.** No compiler, no build step. Each module is two files: `foo.js` (implementation) and `foo.d.ts` (types). `package.json` `types`/`exports` maps point at them directly. JS consumers run the code unchanged; TS consumers pick up the sidecars natively. Revisit when every non-EOL Node LTS supports running `.ts` files natively without a flag. First-class type surface: interfaces for Adapter options, KoaAdapter overlay, batch descriptors, hooks, options bags — all in the `.d.ts` files.
- **Drop CommonJS.** No dual-publish unless the design doc decides otherwise (Node 18+ supports ESM ergonomically; Lambda runtimes 18/20 included).
- **Peer-depend on `@aws-sdk/client-dynamodb` and `@aws-sdk/lib-dynamodb`.** Optional peer on `@aws-sdk/credential-providers` (only used by the client-construction helper, which may move to a separate sub-export or be dropped).
- **Sub-exports in `package.json` `exports` map.** Likely shape: `dynamodb-toolkit` (Adapter), `dynamodb-toolkit/koa` (KoaAdapter), `dynamodb-toolkit/expressions` (patch/filter/projection builders), `dynamodb-toolkit/batch` (batch + transaction runners), `dynamodb-toolkit/mass` (writeList/deleteList/copyList/moveList/readList family), `dynamodb-toolkit/utils` (paths, normalizeFields, subsetObject).
- **Tree-shake friendly.** Each sub-export pulls only its dependencies.

### 6.2 SDK integration

- Replace custom marshalling with `lib-dynamodb` middleware (drop `utils/converter.js`, `utils/convertTypes.js`, `specialTypes`).
- Replace `utils/makeClient.js` + `getProfileName.js` with a thin helper that composes `@aws-sdk/credential-providers`. Or drop entirely and let the consumer construct the client.
- Detect document client by class instance (`client instanceof DynamoDBDocumentClient`) instead of `typeof client.createSet`.
- Update `applyTransaction` chunk size from 25 → **100**; `moveList` chunks from 12 → **50** (50 puts + 50 deletes ≤ 100).
- Adopt v3's `paginateQuery` / `paginateScan` underneath `iterateList`. Keep `paginateList` / `paginateListNoLimit` because `Limit` is pre-filter.
- Audit every `.promise()` call site — none should remain.

### 6.3 Type system (in `.d.ts` sidecars)

Types live in hand-written `.d.ts` files alongside each `.js` module — no `.ts` sources, no compiler (see [[js-plus-dts-no-typescript]]).

- Generic `Adapter<TItem, TKey>` parameterized by item/key types.
- Patch type: `Patch<TItem>` allowing dotted-path keys + the magic delete/separator keys. Investigate accurate typing of nested paths (e.g., `Path<TItem>` recursive type).
- Batch descriptor types: discriminated union by `action`.
- Options bag type for mass ops (`offset`, `limit`, `descending`, `consistent`, `fields`, `filter`).
- `Raw<T>` and `DbRaw<T>` branded marker types.
- Re-export key SDK types so consumers don't import from two trees.

### 6.4 Patches and expressions

- Keep the patch builder. Re-export standalone from `dynamodb-toolkit/expressions`.
- Rename the magic keys to match the article: `__delete` → `_delete`, `__separator` → `_separator`. (See §4.2 #1.)
- Optional: add `_array` patch operations (splice/drop/append) per article §"HTTP verbs: PATCH". Defer unless a consumer asks.
- Decide on `_findOne` / `_findAll` semantics for `PATCH` and `DELETE` — mass ops by criterion, on top of `Adapter.cloneAll` / `deleteAll` with a filter. (See §4.2 #6.)
- Add an explicit `condition()` builder for `ConditionExpression` — currently glued in via the `updateParams` hook.
- Decide whether `searchable` should grow to support tokenized search, prefix match, or stay substring-only.

### 6.5 Mass operations

- Keep all current `*ByParams`, `*ByKeys`, `*All` variants.
- Decide whether `generic*` variants survive in v3. Likely yes, possibly merged into a single `{strategy: 'native' | 'sequential'}` option.
- Add `iterateAll` / `iterateAllByKeys` async iterators yielding un-revived items for streaming jobs.
- Update `applyTransaction` chunk size from 25 → 100 and `moveList` chunks from 12 → 50 (v3 raised the transaction limit).
- **Plumb `needTotal` through to the Adapter as a designer-time option.** The utility supports `{data, offset, limit}` without `total` already; only the high-level layer hardcodes `needTotal: true`. Add `options.needTotal` on `Adapter.getAllByParams` / `getAll` and a KoaAdapter **per-route** option (e.g., on the route-registration helper or as part of the routes config). This is a **designer's decision** — the API designer building on top of the Adapter chooses per endpoint whether `total` is worth its `Select: COUNT` scan. Don't ship a built-in `?total=0` query parameter; a designer _may_ wire one up if their API lets the client choose, but that's their call. `total` can be expensive on large tables, so the design-time knob should exist; turning it into wire surface should not be a default.

### 6.6 KoaAdapter and REST

Keep the REST layer; there is no community successor. Refactor it as a stack of _helpers_ (parsers, envelope builders, error builder) plus a _framework-agnostic core_ with thin Koa/Express/Hono wrappers. Each functional requirement in §4.1 maps to one helper; concrete naming is configurable policy, not hard-coded.

Helper inventory the v3 toolkit should ship (see §4.1 for the full mapping):

- **Parsers**: `parseFields`, `parseSort`, `parseFilter`, `parsePatch`, `parseNames`, `parsePaging`. Each returns a structured result; `KoaAdapter` wires them up but the consumer can call them directly in custom routes.
- **Builders**: `buildPaginationEnvelope`, `buildErrorBody`, `buildItemResponse`. Configurable key names and shapes.
- **Search/filter helpers**: `filterBySubstring` (current `searchable` story) plus `filterByExample` (partial-object equality) for §4.1 #9.
- **Mass by criterion**: `findAndPatch`, `findAndDelete`, `findOneAndPatch`, etc. for §4.1 #4. Built on `Adapter.cloneAll` / `deleteAll` + a filter.
- **Cache helpers**: optional `etag(item)`, `lastModified(item)`, `vary(ctx)` hooks; default Koa middleware emits `304` on a match.
- **Default routing pack**: `routes: 'standard'` mounts `/`, `/:key`, `/-by-names`, `/:key/-clone`, `/:key/-move`, `/-clone`, `/-move`, `/-clone-by-names`, `/-move-by-names`. Method-prefix character is configurable (default `-`).

URL grammar to keep as the default routing pack (consumer can override): `/`, `/:key`, `/-load`, `/-clone`, `/-move`, `/-by-names`, `/-clone-by-names`, `/-move-by-names`, `/:key/-clone`, `/:key/-move`.

Status codes are policy, not gospel:

- **`404` is acceptable** for misses (widely understood in REST, the article's caution is project-specific). Keep as default; configurable.
- **`500` needs a review** — the current `tests/server.js` returns `500` from the catch-all. The toolkit should map known error families to specific codes (`409` consistency, `422` validation, `429` throttling, `503` transient SDK), and only fall back to `500` for genuinely unknown failures, always with a body via the error envelope.

Naming conventions are policy, not gospel. The default ships article-style (`_delete`, `items`, `_findOne`) but each is a single config string. See §4.2.

Open questions: extract to `@dynamodb-toolkit/koa`? Add Express/Hono wrappers via the framework-agnostic core?

### 6.7 Hooks and conventions

- Keep `prepare` / `revive` / `validateItem` / `checkConsistency` / `prepareKey` / `restrictKey` / `prepareListParams` / `updateParams`. Audit names for v3 (e.g., `updateParams` could be `updateInput` to match v3 Command terminology).
- Reconsider `Raw` / `DbRaw`. With `lib-dynamodb` always on, only one bypass shape is needed; likely collapse to a single `Raw<T>` brand.
- Indirect indices: keep, document better.
- Searchable fields: keep. Document `-search-` (DB-side) and `_`-prefix (wire-side) conventions as distinct things in the wiki.
- Technical-field convention: `-` prefix for DB-internal columns (matches current toolkit and is hidden from clients); `_` prefix for wire-level pseudo-properties (matches the article).

### 6.8 Testing

- Replace Postman + Koa loop with a real test runner. Candidate: `tape-six` for parity with sibling projects.
- Layers:
  - **Unit:** expression builders, marshalling boundary, patch logic, path utilities — pure functions, no SDK.
  - **Integration:** Adapter + mass ops against `aws-sdk-client-mock` mocking `DynamoDBDocumentClient`.
  - **End-to-end:** optional, against DynamoDB Local in CI.
- Move `tests/data.json.gz` into a fixtures module; keep Star Wars planets as the canonical example dataset.
- Keep a Postman-flavored example for manual smoke testing the KoaAdapter, but out of the test surface.

### 6.9 Documentation

- Rebuild `wiki/` after the API stabilizes. Most pages still apply conceptually; signatures change.
- Add a v2-toolkit → v3-toolkit migration guide.
- Add an SDK v2 → SDK v3 cheat sheet for consumers.
- Cross-link the article (`web-apps-client-server-api-design.md`) from the KoaAdapter overview as the rationale for REST conventions.

### 6.10 Observability and ops

- Optional `logger` constructor option receiving `{event, command, durationMs, retryCount}`.
- Metrics hooks (CloudWatch EMF? OTel?) — likely "consumer's job" and out of scope, but document the seams.

## 7. What needs a decision before the design doc

4. **`returnRaw` modes — keep three (none/`raw`/`db-raw`) or collapse?**
5. **Patch surface — keep magic keys (renamed to `_delete`/`_separator`) or move to an explicit options bag (`patch(item, {delete: [...], separator: '.'})`)?**

Decision: which defaults match Eugene's house style, and which are left at the article's defaults? Specifically, review the `500`-on-catch-all behavior in `tests/server.js` — map known error families to specific codes (`409`/`422`/`429`/`503`) before falling back to `500`.

10. **Test runner choice — `tape-six` vs `node:test` vs `vitest`?**
11. **Wiki vs in-repo `docs/` vs both?**
12. **Implement `_array` patch operations now or defer?**
